# dev/ts-prune-allowlist.txt — Layer 4.3 ts-prune gate baseline.
#
# One entry per line in the form `path - exportName` (no line number —
# ts-prune emits the column, but matching by line means any edit
# elsewhere in the file would churn the allowlist). Lines starting with
# "#" or blank are ignored.
#
# Add an entry only after verifying that the export is actually
# referenced through indirection ts-prune cannot see — most commonly:
#
#   1. Cross-tsconfig boundaries: tsconfig.web.json (renderer) and
#      tsconfig.node.json (main process + preload) are separate
#      compilation graphs, so an export consumed across the boundary
#      shows up as unused on whichever side declared it. Examples:
#      `SidecarManager`, `SidecarInfo`, `ElectronAPI`.
#
#   2. Re-exported convenience namespaces in api/client.ts (`api`,
#      `Lifecycle`, `Echo`, `*PromptTemplate*`) — kept available for
#      callers even when no current component imports them.
#
#   3. Internal-only types/interfaces in api/client.ts and
#      stores/appStore.ts that ts-prune flags as "used in module"
#      because they are referenced exclusively within their declaring
#      file. They are still exported in case a sibling component
#      starts importing the type later.
#
#   4. Domain types in types/safety.ts and types/usage.ts that are
#      consumed via property access on a parent type rather than
#      directly imported by name.
#
# When you delete dead code, remove the corresponding allowlist line.
